## 🎯 Этап 1: Модели данных и миграция (2 дня)

### Цель
Обновить Prisma схему и мигрировать существующие данные без потерь

### Задачи

#### День 1

- [ ] **Создать новые модели в `prisma/schema.prisma`**
  - `KakeboCategory` (системные + пользовательские)
  - `KakeboMonthlyGoal`
  - `KakeboFixedExpense`
  - `KakeboMonthlyBudget`
  
- [ ] **Изменить `KakeboEntry`**
  - Заменить `cuid()` на `uuid()`
  - Добавить `categoryId` (nullable для миграции)
  - Добавить `categoryOld` для временной совместимости
  - Добавить `isAutoGenerated`

- [ ] **Обновить `KakeboSettings`**
  - Добавить `income` поле

- [ ] **Создать миграцию:**
  ```bash
  npx prisma migrate dev --name add_kakebo_v2_models
  ```

#### День 2

- [ ] **Создать скрипт миграции данных**
  - `scripts/migrate-kakebo-to-v2.ts`
  - Создать системные категории (LIFE, CULTURE, EXTRA, UNEXPECTED)
  - Перенести все записи в новые категории
  - Проверить целостность данных

- [ ] **Запустить миграцию на dev-окружении**
  ```bash
  node scripts/migrate-kakebo-to-v2.ts
  ```

- [ ] **Проверить результаты**
  - Все записи имеют `categoryId`
  - Нет NULL значений
  - Системные категории созданы

- [ ] **Обновить `Profile` модель**
  - Добавить relations к новым моделям

### Критерии готовности

- ✅ Все новые модели в схеме
- ✅ Миграция применена без ошибок
- ✅ Данные перенесены корректно
- ✅ Нет потерь записей
- ✅ Все ID имеют UUID формат

### Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Потеря данных | Низкая | Бэкап перед миграцией |
| Ошибки в миграции | Средняя | Тест на dev-окружении |
| Долгая миграция | Низкая | Оптимизация SQL запросов |

## 🎯 Этап 3: Бэкенд-логика автосоздания записей (1 день)

### Цель
Автоматически создавать записи из фиксированных трат

### Задачи

#### День 5

- [ ] **Создать сервис для генерации записей**
  ```typescript
  // server/services/kakebo.service.ts
  export async function generateFixedExpensesForMonth(
    profileId: string, 
    year: number, 
    month: number
  ): Promise<void>
  ```

- [ ] **Логика генерации**
  - Найти все активные фиксированные траты
  - Для каждой создать запись в `KakeboEntry`
  - Установить `isAutoGenerated = true`
  - Пропускать если запись уже существует

- [ ] **Триггеры**
  - При создании `KakeboFixedExpense` — генерировать на 4 месяца вперёд
  - При обновлении `KakeboFixedExpense` — опционально пересоздать
  - При входе пользователя в новый месяц — проверить пропущенные

- [ ] **Обработка ошибок**
  - Если категория удалена — не создавать запись
  - Если профиль заблокирован — не создавать

- [ ] **Тестирование**
  - Создать фиксированную трату
  - Проверить что записи созданы
  - Удалить фиксированную трату
  - Проверить что записи не удалились (только пометка)

### Критерии готовности

- ✅ Фиксированная трата создаёт 4 записи
- ✅ Записи помечены `isAutoGenerated = true`
- ✅ Нет дубликатов
- ✅ Обрабатываются ошибки

### Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Дубликаты записей | Средняя | Проверка существования |
| Проблемы с транзакциями | Средняя | Использовать `$transaction` |
| Производительность | Низкая | Пакетная обработка |
