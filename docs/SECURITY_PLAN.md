# Раздел 3. Обеспечение безопасности веб-приложений

Веб-приложение Экономикус, как и любая образовательная платформа, подвержено различным угрозам безопасности. Источники угроз можно разделить на внутренние и внешние.

## 3.1. Источники угроз безопасности

### Внутренние угрозы

| Источник угрозы | Описание | Потенциальные последствия |
|-----------------|----------|---------------------------|
| Неавторизованный доступ сотрудников | Сотрудники, имеющие доступ к панели автора или админ-панели, могут случайно или намеренно изменить или удалить данные курсов | Утрата или искажение образовательного контента |
| Утечка данных через резервные копии | Несанкционированный доступ к файлам резервных копий базы данных | Компрометация персональных данных пользователей |
| Слабые пароли | Использование простых или стандартных паролей для входа в аккаунт | Взлом учётной записи пользователя |
| Ошибки конфигурации | Неправильная настройка прав доступа к файлам и папкам | Возможность чтения или изменения критических файлов |
| Ошибки авторизации на клиенте | Обход проверок авторизации через React Router без серверной проверки | Несанкционированный доступ к защищённым страницам |

### Внешние угрозы

| Источник угрозы | Описание | Потенциальные последствия |
|-----------------|----------|---------------------------|
| SQL-инъекции | Внедрение злонамеренных SQL-запросов через формы ввода или URL-параметры | Получение доступа к базе данных, утечка или модификация данных пользователей |
| XSS-атаки (межсайтовое скриптование) | Вставка вредоносного JavaScript-кода в комментариях, отзывах или профилях пользователей | Кража сессионных данных, перенаправление на фишинговые сайты |
| CSRF-атаки (подделка межсайтовых запросов) | Выполнение нежелательных действий от имени авторизованного пользователя | Изменение профиля, удаление избранного, модификация прогресса |
| DDoS-атаки | Перегрузка сервера большим количеством запросов | Недоступность сайта для легитимных пользователей |
| Эксплойты уязвимостей | Использование известных уязвимостей в Node.js, Prisma, React или зависимостях | Компрометация сервера, удалённое выполнение кода |
| Брутфорс (подбор паролей) | Автоматизированный подбор пароля к аккаунтам пользователей | Несанкционированный доступ к личным кабинетам |
| Межсайтовая утечка данных | Утечка данных через CORS или неправильные заголовки | Получение доступа к данным других пользователей |

## 3.2. Методы защиты доступа к данным и защиты кода

Для защиты веб-приложения от перечисленных угроз применены следующие методы:

### Защита доступа к данным

| Метод | Реализация в Экономикус |
|-------|-------------------------|
| Шифрование передачи данных | HTTPS (SSL-сертификат) — конфигурация на стороне хостинга |
| Шифрование паролей | Пароли пользователей хранятся в базе данных в захешированном виде с использованием bcryptjs (12 раундов) |
| Ролевая модель доступа | Разграничение прав: USER, AUTHOR, MODERATOR, ADMIN. Доступ к /admin, /author только для авторизованных с проверкой роли через requireAdmin() и requireRole() |
| Безопасное хранение конфиденциальных данных | Файл .env с паролями доступа исключён из Git-репозитория (добавлен в .gitignore) |
| Защита сессий | HttpOnly cookies с атрибутами SameSite=Strict и Secure; срок действия — 30 дней |
| Проверка блокировки | При аутентификации проверяется поле isBlocked; заблокированные пользователи не могут войти |
| Защита от брутфорса | При неудачной попытке входа используется стандартная задержка; отсутствие подробных сообщений об ошибках (общее сообщение "Неверный email или пароль") |

### Защита кода

| Метод | Реализация в Экономикус |
|-------|-------------------------|
| Защита от SQL-инъекций | Использование Prisma ORM для всех запросов к базе данных; Prisma автоматически экранирует параметры |
| Защита от XSS | React по умолчанию экранирует вывод; использование dangerouslySetInnerHTML только с проверенным контентом |
| Защита от CSRF | SameSite=Strict атрибут в cookies; токены не требуются благодаря SameSite |
| Валидация данных | Использование Zod для валидации входных данных на сервере |
| Защита API | Все защищённые роуты проверяют авторизацию через middleware requireAuth |
| Проверка авторизации на сервере | Каждый защищённый эндпоинт API проверяет сессию; client-side ProtectedRoute не является единственной защитой |
| Ограничение доступа к файлам | Статические файлы обслуживаются корректно; критические маршруты недоступны без авторизации |

### Защита инфраструктуры

| Метод | Реализация |
|-------|------------|
| Изоляция зависимостей | node_modules не входят в репозиторий; используется .gitignore |
| Безопасная конфигурация | Все переменные окружения хранятся в .env, исключённом из Git |
| Разделение окружений | Разные конфигурации для development и production |
| Очистка сессий | Автоматическое удаление истекших сессий (cleanupExpiredSessions) |

## 3.3. Диагностика наличия вредоносных программ

Для диагностики и выявления вредоносных программ используются следующие методы и инструменты:

### Сканирование зависимостей

Проект использует минимальное количество внешних зависимостей. Все зависимости указаны в package.json и устанавливаются из официального npm-реестра. Риск внедрения вредоносного кода через зависимости контролируется следующим образом:

- Регулярные обновления зависимостей через npm audit
- Использование проверенных библиотек с активной поддержкой
- Отсутствие сторонних CDN для скриптов

### Анализ логов сервера

На сервере (при деплое) доступны логи доступа и ошибок. Регулярный анализ логов позволяет выявить:

- Подозрительные запросы (попытки SQL-инъекций, XSS)
- Аномальное количество запросов (признак DDoS)
- Нестандартные User-Agent
- Множественные попытки авторизации с разными email

### Сканирование уязвимостей

Для периодической проверки безопасности используются:

| Инструмент | Назначение |
|------------|------------|
| npm audit | Проверка уязвимостей в зависимостях |
| SecurityHeaders.io | Проверка наличия и корректности HTTP-заголовков безопасности |
| SSL Labs | Проверка качества SSL-сертификата и поддержки TLS |
| Google Lighthouse | Проверка безопасности, HTTPS, корректной конфигурации |

## 3.4. Мероприятия по защите приложения

### Реализованные мероприятия

| Мероприятие | Статус | Описание |
|-------------|--------|----------|
| HTTPS | Выполнено | SSL-сертификат на хостинге |
| Защита сессий (CSRF) | Выполнено | HttpOnly cookies с SameSite=Strict |
| Защита от SQL-инъекций | Выполнено | Использование Prisma ORM |
| Хеширование паролей | Выполнено | bcryptjs, 12 раундов |
| Валидация данных | Выполнено | Zod на сервере, React Hook Form + Zod на клиенте |
| Ролевая модель | Выполнено | requireAuth, requireRole, requireAdmin middleware |
| Проверка блокировки | Выполнено | isBlocked проверяется при аутентификации |
| Исключение конфиденциальных файлов | Выполнено | .env в .gitignore |
| Защита API | Выполнено | Все защищённые роуты проверяют авторизацию |
| Очистка сессий | Выполнено | cleanupExpiredSessions() |

### Рекомендуемые мероприятия

| Мероприятие | Приоритет | Описание |
|-------------|-----------|----------|
| Настройка Content-Security-Policy (CSP) | Высокий | Ограничение источников загрузки скриптов и стилей для защиты от XSS |
| Двухфакторная аутентификация (2FA) | Средний | Дополнительная защита входа в аккаунт |
| Rate limiting | Средний | Ограничение частоты запросов для защиты от DDoS и брутфорса |
| Логирование действий | Средний | Ведение журнала изменений курсов, удалений контента |
| HTTP-заголовки безопасности | Средний | X-Content-Type-Options, X-Frame-Options, Referrer-Policy |
| Регулярный аудит безопасности | Низкий | Периодическая проверка уязвимостей с помощью специализированных инструментов |
| Выпуск версий зависимостей | Низкий | Регулярное обновление @hono, @prisma/client, react и других пакетов |

## 3.5. Выводы о безопасности

Веб-приложение Экономикус соответствует базовым требованиям безопасности веб-приложений:

| Критерий | Оценка |
|----------|--------|
| Защита передачи данных | HTTPS, защищённые cookies |
| Защита аутентификации | Хеширование паролей bcryptjs, HttpOnly cookies |
| Защита от SQL-инъекций | Prisma ORM |
| Защита от XSS | React экранирование по умолчанию |
| Защита от CSRF | SameSite=Strict cookies |
| Защита API | Middleware requireAuth на всех защищённых роутах |
| Ролевая модель | requireRole, requireAdmin |
| Конфиденциальность файлов | .env исключён из Git |

### Основные риски, требующие внимания:

- Отсутствие Content-Security-Policy (CSP) — рекомендуется настроить для защиты от XSS-атак
- Отсутствие rate limiting — рекомендуется ограничить частоту запросов
- Нет логирования действий пользователей — рекомендуется внедрить для отслеживания изменений
- Отсутствие двухфакторной аутентификации — для повышения безопасности входа

### Особенности учебного проекта:

Поскольку Экономикус является учебным (дипломным) проектом, применяются следующие допущения:

- Реальные платежи не обрабатываются
- Персональные данные хранятся в тестовом режиме
- SSL-сертификат настраивается на уровне хостинга
- Полноценный аудит безопасности требует промышленного деплоя

**Общая оценка:** реализованные меры защиты обеспечивают достаточный уровень безопасности для учебного веб-приложения. Рекомендованные мероприятия позволят дополнительно усилить защиту и минимизировать риски при переходе к промышленной эксплуатации.
